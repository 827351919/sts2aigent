# AI_RULES.md

## 项目 AI 协作规则

本文档用于约束 Codex、Claude 或其他 AI 编码代理在本项目中的行为，避免项目在早期阶段偏离目标。

## 1. 总体原则

### 1.1 闭环优先

- 当前第一目标是打通最小自动运行闭环
- 任何实现都应优先服务于：
  - 读取状态
  - 生成合法动作
  - 执行动作
  - 刷新状态并继续

### 1.2 先简单后复杂

- 先实现保守可运行版本
- 不要在项目早期引入过多抽象层
- 不要为了“架构好看”牺牲落地速度

### 1.3 规则优先于 LLM

- 明确合法性约束的逻辑优先写成规则
- LLM 只用于规则难以覆盖的决策补充
- 不允许将动作索引、目标合法性等关键约束完全交给 LLM 猜测

### 1.4 先 HTTP，后 MCP 抽象

- 当前阶段默认直连 `STS2MCP` 的 HTTP API
- 如非必要，不提前实现复杂 MCP transport 层

## 2. 实施约束

### 2.1 文件修改前先读上下文

修改代码前，至少先读取：

- `docs/PROJECT_STATUS.md`
- `docs/NEXT_ACTIONS.md`
- `docs/DECISIONS.md`
- `docs/PRD.md`

### 2.2 每次只推进一个明确目标

优先选择以下类型的任务：

- 打通一个模块
- 补齐一个状态 handler
- 引入一类知识数据
- 修复一个明确 bug

避免一次同时推进：

- 大量重构
- 多个未验证子系统
- 与当前阶段无关的长期功能

### 2.3 最小必要改动

- 只改和当前任务直接相关的文件
- 不做无关命名整理
- 不做无关重构
- 不顺手重写大块代码，除非原设计明显阻塞当前目标

### 2.4 未知状态必须安全处理

- 遇到未知 `state_type` 时，不允许默认乱执行动作
- 必须进入 safe handler：
  - 记录日志
  - 保存原始状态
  - 选择等待、退出或请求人工介入

### 2.5 动作执行必须二次校验

所有动作执行前应检查：

- 当前 `state_type` 是否匹配
- 目标 index 是否合法
- target 是否存在
- 当前状态是否已经变化

## 3. 状态机规则

### 3.1 核心状态优先级

当前优先实现顺序：

1. `monster` / `elite` / `boss`
2. `combat_rewards`
3. `card_reward`
4. `map`
5. `event`
6. `rest_site`
7. `shop`
8. `treasure`
9. `card_select`
10. `relic_select`
11. `hand_select`

### 3.2 动作后必须刷新状态

- 任何动作执行后，默认重新获取最新状态
- 不要依赖旧索引连续执行多个动作，除非逻辑已明确保证安全
- 对战斗和奖励场景尤其要谨慎

### 3.3 未推进不等于成功

- 接口返回成功，不代表游戏状态一定推进
- 必须通过 re-fetch 判断状态是否真的变化

## 4. 知识使用规则

### 4.1 AIBOT 的使用方式

- 优先使用其静态知识：
  - cards
  - relics
  - enemies
  - events
  - builds
- 其次借鉴其启发式思路
- 最后才考虑迁移具体运行时代码

### 4.2 知识不是绝对真理

- AIBOT 知识是参考，不是不可质疑的标准答案
- 当前局面、当前血量、当前路径风险可能高于静态知识偏好

## 5. LLM 使用规则

### 5.1 何时允许调用 LLM

- 多个候选动作接近，规则无法可靠区分
- 复杂战斗局面需要额外推理
- 当前场景已有结构化上下文与知识摘要可供输入

### 5.2 何时不应该调用 LLM

- 当前只有一个合法动作
- 明显是索引类、确认类、推进类操作
- 当前状态高频快速变化，LLM 可能来不及返回
- LLM 不可用时已有规则兜底

### 5.3 LLM 输出必须受限

- 必须转换为结构化动作
- 必须通过 schema 校验
- 必须通过本地合法性校验
- 失败时必须回退到规则策略

## 6. 日志与复盘规则

### 6.1 必须记录的内容

- 当前 `state_type`
- 关键状态摘要
- 决策结果
- 决策来源：
  - heuristic
  - knowledge
  - llm
  - fallback
- 动作执行结果
- 错误信息

### 6.2 出问题时优先保留证据

如果遇到卡死、未知状态或执行失败，优先：

1. 记录原始状态
2. 记录当前决策
3. 记录动作响应
4. 再决定是否重试

## 7. 当前阶段禁止事项

- 不要优先做 Web UI
- 不要优先做多人模式
- 不要优先做训练/学习系统
- 不要过早支持过多模型提供商
- 不要把目标重新扩大成“大而全平台”

## 8. 文档维护规则

- 完成关键里程碑后更新 `PROJECT_STATUS.md`
- 完成任务后同步调整 `NEXT_ACTIONS.md`
- 出现新技术取舍时立即记录到 `DECISIONS.md`
- 若实现和 `PRD.md` 偏离，必须更新文档或说明原因
